Handoff técnico — Módulo Mascotas

Resumen para Rafa de lo construido en esta tanda: módulo de Mascotas completo, listo para integrar el host de fotos. Local-first (IndexedDB), Next.js 16 + React 19 + Tailwind v4, Convex como convergencia opcional (apagada).

✅ tsc + build limpios 🚀 Desplegado en producción 🐾 Sin brazaletes 🔗 Custodia append-only

1 · Qué se construyó

Módulo /mascotas análogo a Niños e Insumos. Cada mascota se registra con foto, datos y ubicación (geografía real de Venezuela) y recibe un código interno MAS-XXXX. Su historial de movimiento es una cadena de custodia append-only (mismo rigor que Niños): quién la trasladó, a quién se entregó, ingreso/salida de refugio y atención veterinaria — con regla de dos personas y confirmación del receptor (R3), inmutable (R10). Incluye:
  • 3 pestañas: Mascotas (fichas), Cartelera (se busca / encontrada / reunificada) y Necesidades.
  • Refugio como campos en la ficha (residencial / público acondicionado / institucional / hogar temporal).
  • Necesidades veterinarias (medicinas_vet, atencion_vet, operacion, alimento_mascota, hospedaje, traslado_mascota) que reutilizan el motor de Insumos (tabla reportes + matching). Cada necesidad puede ir atada a una ficha (mascotaId) y aparece también en /insumos. Los proveedores entran como ofertas.
  • Cartel imprimible/compartible (foto + datos + zona + contacto + QR del código) y hoja de chapas/collar QR opcional.
  • Datos de ejemplo sembrables desde la UI (botón "Cargar datos de ejemplo").

3 · Integración pendiente: host de fotos (Cloudflare)

Decisión: el host definitivo lo defines tú (Cloudflare R2 / Images). Dejé el código con un único punto de integración: src/lib/fotos.ts → subirFotoPublica(blob). Hoy devuelve null (sin host) y la app cae al respaldo: foto local (offline-first, nunca sale del teléfono, R7) + campo para pegar una URL pública a mano. La UI ya llama a esa función en el alta de mascota — solo hay que rellenar su cuerpo y que subidaConfigurada() dé true.
  • Contrato: entra un Blob, sale la URL pública permanente (CDN) o null.
  • Sugerido leer la config de NEXT_PUBLIC_FOTO_UPLOAD_ENDPOINT para no acoplar a un proveedor.
  • Opciones: (a) R2 + Worker que firma un PUT y devuelve la URL pública; (b) Cloudflare Images con "direct creator upload URL" y usar la variante de entrega como fotoUrl.
  • fotoUrl (string) SÍ sincroniza a Convex y es lo que hace el cartel compartible; fotoBlob nunca viaja.

4 · Heads-up técnico: upgrade de IndexedDB v4 → v5

Subí DB_VERSION de 4 a 5 para crear los object stores nuevos. En navegadores que ya tenían la base abierta en v4 (pestaña vieja de la app, o uso previo) el open(5) queda bloqueado esperando que se cierren las conexiones v4 → la app se queda en "Cargando…". En un navegador nuevo arranca en v5 y funciona. Lo confirmamos en pruebas: pasó en un perfil con datos viejos.

Fix sugerido (en src/lib/db.ts → abrir()): manejar el bloqueo y reaccionar a cambios de versión, p.ej.: 
req.onblocked = () => { /* avisar: cierra otras pestañas de Waraira */ };
// y en cada conexión abierta:
db.onversionchange = () => db.close();  // libera para que el upgrade entre
Alternativa rápida para probar en un equipo con datos viejos: borrar la base (DevTools → Application → IndexedDB → waraira → Delete) y recargar. Ojo: borra los datos locales de ese equipo.
